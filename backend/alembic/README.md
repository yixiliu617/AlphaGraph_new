# Alembic migrations — AlphaGraph

Run from the project root:

```bash
# Apply latest migrations
alembic -c backend/alembic.ini upgrade head

# Generate a new migration after changing ORM models
alembic -c backend/alembic.ini revision -m "describe change" --autogenerate

# Show current head + history
alembic -c backend/alembic.ini current
alembic -c backend/alembic.ini history --verbose

# Roll back one revision
alembic -c backend/alembic.ini downgrade -1
```

The DB URL is taken from `backend/app/core/config.py:settings.POSTGRES_URI`,
which defaults to `sqlite:///./alphagraph.db` for dev. To target the local
Postgres container set:

```
POSTGRES_URI=postgresql+psycopg2://alphagraph:alphagraph_dev@localhost:5432/alphagraph
```

in your `.env`.

## Migration file conventions

- File names: `0001_create_user_alert_heartbeat.py` — numeric prefix in
  4-digit hex-friendly form, snake_case description after.
- Each migration has both `upgrade()` and `downgrade()` populated. Don't
  ship one-way migrations; prod recovery sometimes needs the reverse.
- Don't put data backfills inside schema migrations. Use a separate
  `backend/scripts/migrations/<rev>_data.py` if needed.

## When you add a new ORM model

1. Define the model in `backend/app/models/orm/<name>_orm.py`.
2. Import it in `backend/app/db/session.py` AND in `alembic/env.py`
   (the comment block lists the existing imports).
3. `alembic revision -m "..." --autogenerate` will diff the live schema
   against the metadata and emit a migration draft.
4. **Always review** the autogenerated migration before applying — Alembic's
   diff is good but not perfect for renames, type changes, or
   server-side defaults.
